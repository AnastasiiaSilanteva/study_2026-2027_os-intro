Контрольные вопросы

Что такое системы контроля версий (VCS) и для решения каких задач они предназначаются?

VCS — программные средства для учёта изменений файлов, совместной работы над проектами, хранения истории версий, возможности отката к предыдущим состояниям, разрешения конфликтов при одновременном редактировании. Объясните понятия VCS: хранилище, commit, история, рабочая копия.

Хранилище (repository) — место хранения всех версий файлов и служебной информации. Commit (фиксация) — сохранение набора изменений в репозиторий с комментарием. История (history) — последовательность коммитов, позволяющая проследить эволюцию проекта. Рабочая копия (working copy) — локальная версия файлов проекта, с которой непосредственно работает пользователь. Централизованные и децентрализованные VCS: различия и примеры.

Централизованные (CVS, Subversion): единый сервер с репозиторием, клиенты получают рабочие копии. Необходимо подключение к серверу для большинства операций. Децентрализованные (Git, Mercurial): каждый клиент имеет полную копию репозитория (включая историю), можно работать локально, синхронизация с другими репозиториями по необходимости. Действия с VCS при единоличной работе с хранилищем.

Инициализация репозитория (git init). Добавление файлов под версионный контроль (git add). Фиксация изменений (git commit). Просмотр истории (git log), сравнение версий (git diff). Создание веток (git branch), слияние (git merge). Порядок работы с общим хранилищем VCS.

Клонирование удалённого репозитория (git clone). Получение актуальных изменений (git pull). Внесение изменений в локальной ветке. Фиксация изменений (git commit). Отправка изменений в удалённый репозиторий (git push). При необходимости разрешение конфликтов.

Основные задачи, решаемые git.

Управление версиями файлов. Поддержка ветвления и слияния. Распределённая работа без постоянного подключения к серверу. Обеспечение целостности данных (через хеширование). Возможность подписи коммитов для подтверждения авторства.

Назовите и кратко охарактеризуйте основные команды git.

git init — создание репозитория. git clone — копирование удалённого репозитория. git add — добавление изменений в индекс. git commit — фиксация индекса с комментарием. git status — состояние рабочей копии. git log — история коммитов. git diff — различия между версиями. git branch — управление ветками. git checkout — переключение между ветками/коммитами. git merge — слияние веток. git pull — получение изменений из удалённого репозитория. git push — отправка изменений в удалённый репозиторий. git remote — управление удалёнными репозиториями.

Приведите примеры использования при работе с локальным и удалённым репозиториями.

Локальный: git init, git add file.txt, git commit -m "Add file", git log. Удалённый: git clone ..., git pull, git push, git remote -v.

Что такое ветви (branches) и зачем они нужны?

Ветви позволяют разрабатывать новые функции или исправления изолированно от основной линии разработки. После завершения работы ветку можно слить с основной. Это упрощает параллельную работу нескольких разработчиков и эксперименты.

Как и зачем можно игнорировать некоторые файлы при commit?

Через файл .gitignore перечисляются шаблоны имён файлов/каталогов, которые Git не должен добавлять в репозиторий. Это нужно для исключения временных файлов, результатов сборки, конфиденциальных данных, чтобы они не загромождали репозиторий и не попадали в общий доступ.
